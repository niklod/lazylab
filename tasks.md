- [ ] Background cache refresh must trigger UI re-render (reverses ADR 009)
  - **Rationale / repro:** user opens the app → stale MR list served from cache + silent background refresh → fresh list now in memory+disk, but nothing on screen changes → new MRs created upstream stay invisible until the user fully exits and relaunches (and even then only if the TTL has elapsed). This defeats the stale-while-revalidate contract: SWR promises "show stale, swap to fresh automatically", and the current Go port truncates it to "show stale, swap on next explicit user action". The original Python cache did fire `_on_refresh` for exactly this reason (ADR 006); ADR 009 dropped it on a UX-flicker argument that in practice costs the user real content.
  - **DoD:**
    1. `internal/cache/cache.Cache` exposes a refresh notification channel — exact API to be decided in the fix (candidates: `OnRefresh(func(namespace, key string))` single-subscriber callback, or `Subscribe() <-chan RefreshEvent` multi-subscriber channel). Fires **only** on background refresh that actually changed memory+disk state (hit in `refreshIfPresent`), not on synchronous misses and not on no-op refreshes.
    2. TUI wires the callback once at startup in `cli.Run` → fans out to active views (`ReposView`, `MRsView`, `DetailView` sub-renderers) via `g.Update`. Only views whose currently-displayed namespace matches the fired event re-render; an event on `mr_pipeline` must not force the repos pane to repaint.
    3. No content shift / cursor jump on re-render — views must preserve cursor position (match by IID/project ID), search query, scroll offset, and focus.
